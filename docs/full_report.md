# Russian ASR ITN Hybrid — полный отчёт

## Обзор проекта

Обратная текстовая нормализация (ITN, Inverse Text Normalization) для русского
языка в контексте ASR: преобразование числительных, записанных словами, в
цифровую форму. Пример: `двести пятьдесят рублей` → `250 рублей`.

Проект реализует **два параллельных подхода**:
1. **Rule-based парсер** (словари + mag-based state machine) — 99.8% на
   calibration.f
2. **ruT5-small + LoRA** (seq2seq) — 57.3% exact accuracy на clean synthetic

Оба подхода объединены в **гибрид**: парсер по умолчанию, T5 при низкой
уверенности (отключён, требует дообучения на noisy).

## 2. Rule-based парсер

### 2.1 Архитектура

```
Входной текст
    │
    ▼
_asr_preprocess() — regex-коррекция ASR-ошибок (мягкий знак, "двеси" compound)
    │
    ▼
Токенизация по пробелам
    │
    ▼
┌──────────────────────────────────────┐
│           Два параллельных пути      │
│                                      │
│  Path A (legacy, default):           │
│    normalize_text()                  │
│    → lookup_word()                   │
│    → parse_number_group() (mag)      │
│                                      │
│  Path B (sequence, --parser-type):   │
│    normalize_text_sequence()          │
│    → classify() (token_classifier)   │
│    → parse_sequence() (state machine)│
└──────────────────────────────────────┘
    │
    ▼
Результат: числа цифрами + остальной текст
```

### 2.3 Sequence parser (token_classifier.py + sequence_parser.py)

Замена ad-hoc парсера на систематический token classifier + state machine.

**`src/token_classifier.py`** — единый классификатор:
- TokenClass dataclass (value, mag, subtype, raw, confidence)
- Root-based regex patterns (33 корня, сортировка по убыванию длины)
- Exact dict match → root match → ordinal detection → fused compound → vague
- `_NUMERAL_DICT` из словарей + ASR_ERRORS + дополнительные формы
- Vague "тыщ" context heuristic

**`src/sequence_parser.py`** — state machine:
- Состояния: START → ACCUM → MULTIPLY → enum/ordinal/flush
- Lookahead heuristic для "два два" (enum при одинаковом mag)
- Работает с TokenClass, не с кортежами

**Результаты:**

| Датасет | Current | Sequence | Δ |
|---|---|---|---|
| calibration.f | 99.80% | 99.80% | = |
| synthetic clean | 95.87% | 95.87% | = |
| synthetic noisy | 2.93% | 4.26% | **+1.33%** |
| real.f | 55.23% | 55.23% | = |

**Diff synthetic.f (860 строк):** 290 улучшений (fused compounds), 1 регрессия
("двесте" — исправлена).

### 2.4 Engine ordering (regex priority)

Порядок regex-паттернов в классификаторе критичен для корректной работы:

1. Exact dict match (O(1), быстрый путь)
2. ASR error correction словарь
3. Root-based regex (33 корня, от длинных к коротким)
4. Ordinal suffix detection
5. Fused compound split

Паттерны сортируются по убыванию длины корня: "пятидесят" (7) раньше
"десят" (5), чтобы "пятидесятых" → 50, а не 10.

| Компонент | Файл | Описание |
|---|---|---|
| Token classifier | `src/token_classifier.py` | Единый классификатор, TokenClass |
| Sequence parser | `src/sequence_parser.py` | State machine над TokenClass |
| Legacy parser | `src/parser.py` | Старый parse_number_group() (unchanged) |
| Lexicon | `src/lexicon.py` | lookup_word → classify() (адаптирован) |
| Normalizer | `src/normalizer.py` | normalize_text() + normalize_text_sequence() |
| Disambiguate | `src/disambiguate.py` | pymorphy2 + контекст для сто/три/сорок |
| CLI | `src/cli.py` | --parser-type, evaluate, errors, run |
| EDA | `src/eda.py` | Статистика: accuracy, clean/noisy split |

## 3. ML подход (ruT5 + LoRA)

### 3.1 Проблема

Полное fine-tuning ruT5-small (65M) на <500 samples → 100% переобучение.
После 150 samples на 1 эпоху: eval_loss=0.2, accuracy=0%, мусорные токены
(`<0x57>`). Причина: метки содержали `<pad>` (id=0) вместо `-100`,
модель училась генерировать паддинг.

### 3.2 LoRA (Low-Rank Adaptation)

Обучается только 2.1M параметров (3.2%) вместо 65M.

```python
LoraConfig(
    r=8,           # ранг адаптера
    lora_alpha=16, # масштаб обновлений
    target_modules=['q', 'v', 'k', 'o', 'wi', 'wo'],
    lora_dropout=0.1,
)
```

### 3.3 Результаты обучения

| Запуск | Эпохи | Сэмплы | Accuracy | Num_acc | CER | Время |
|---|---|---|---|---|---|---|
| ep3_s1000 | 3 | 1000 | 0.00% | 0.00% | 0.819 | ~12 мин |
| ep3_s2000 | 3 | 2000 | 2.00% | 1.80% | 0.704 | ~12 мин |
| ep5 | 5 | 4890 | 27.40% | 26.70% | 0.334 | 15.5 мин |
| ep10 | 10 | 4890 | 42.33% | 46.52% | 0.237 | 31 мин |
| ep10_r16 | 10 | 4890 | 43.56% | — | — | 31.3 мин |
| **ep20** | **20** | **4890** | **57.26%** | **60.07%** | **0.152** | **61.6 мин** |

**Динамика обучения (ep20):**

| Эпоха | eval_loss | Accuracy | CER |
|---|---|---|---|
| 1 | 1.3336 | 7.57% | 0.5703 |

| 20 | 0.4848 | 57.26% | 0.1521 |

### 3.4 Проблема `<0x57>` (мусорные токены)

**Причина:** ошибка в формировании `labels`. В `ITNDataset.__getitem__`
метки создавались через `tokenizer(..., padding='max_length')`, что
заполняло пустые позиции токеном `<pad>` (id=0). Модель училась
предсказывать `<pad>` как валидный токен.

**Исправление:** `labels[labels == 0] = -100` (loss игнорирует -100).

### 3.5 Ограничения

- Использованы только clean-данные (29% от доступных)
- LoRA rank=8 (0.9M/66M = 1.36%) — возможно, мало
- ruT5-small (65M) — не ruT5-base (250M)
- Только cardinal + ordinal, нет дробей, дат, процентов
- Нет label smoothing
- Плато после 15 эпох

## 4. Гибридный подход

### 4.1 Архитектура

```
Входной текст
    │
    ▼
┌──────────────┐
│  Парсер      │ → normalize_text() → pred_rule + confidence
│  (rule-based)│
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Оценка      │ → confidence ≥ 0.95 → сразу ответ
│  уверенности │
│  парсера     │ → confidence < 0.75 → fallback на ruT5
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  ruT5-small  │ → model.generate() → pred_t5
│  (LoRA r=8)  │
└──────────────┘
```

### 4.2 Confidence rules

| Признак | Вес |
|---|---|
| has_asr_error ("двеси", "тыщ") | -0.3 |
| has_merged_tokens ("дветысячи") | -0.2 |
| has_unknown_words | -0.2 |
| numeral_density > 3 | -0.1 |
| magnitude_gap (ед→тыс без сотен) | -0.15 |

### 4.3 Результат гибрида

| Мера | Парсер | Гибрид |
|---|---|---|
| calibration.f | 97.6% | 94.4% |
| Время на 500 строк | ~0.2 сек | ~30-60 сек |

Гибрид уступает парсеру — ruT5 не обучена на реальных ASR-паттернах.
Флаг `--hybrid` доступен, не рекомендуется по умолчанию.

## 5. Анализ ошибок

### 5.1 calibration.f: единственная ошибка (99.8%)

```
TASK: дваста рублей скидка на товар если сделать заказ до трёх часов дня
GT:   дваста рублей скидка на товар если сделать заказ до 3 часов дня
PRED: 200 рублей скидка на товар если сделать заказ до 3 часов дня
```

GT содержит ASR-ошибку "дваста" (не нормализовано). Парсер математически
прав — это ITN, а не копирование ASR.

### 5.2 real.f: 55.23% (95/172)

Реальные тексты длиннее (123 символа vs 30), с пунктуацией, дефисами,
процентами, скобками, смешанным языком.

| Категория | Кол-во | % | Пример |
|---|---|---|---|
| Пунктуация прилипла к токену | 17 | 22% | `девять)`, `пятьдесят%` |
| Десятки+единицы разбиты пунктуацией | 11 | 14% | `тридцать девять)` → `девять)` не опознан |
| Дефисные конструкции | 9 | 12% | `фаб-пятьсот`, `тридцать-х` |
| Decimal comma со словом | 7 | 9% | `один,5`, `пять,4` |
| Time-формат | 5 | 6% | `двенадцать:19` |
| Контекстные false positives | ~10 | 13% | `первым`→1, `двух`→2, `трое`→3 |
| Прочее | ~18 | 23% | словоформы, смешанный язык, 41а-семь |

**Корень:** токенизатор режет по пробелам, пунктуация прилипает к слову.
Классификатор не видит `девять)` как "девять".

**Потолок:** ~85-90% после фиксов препроцессинга (отделение пунктуации,
decimal/time regex, стоп-лист контекстных слов).

### 5.3 synthetic.f: 29.48% (4899/16618)

**Взвешенное среднее:**

| Подмножество | Строк | Доля | Accuracy |
|---|---|---|---|
| clean | 4,890 | 29.4% | **95.87%** |
| noisy | 11,728 | 70.6% | **1.80%** |

На clean — отличная точность (96%). На noisy — 1.8%, потому что ASR-шум
ломает слова до неузнаваемости: `тсыячи` (тысячи), `чтыреста` (четыреста),
`трста` (триста), `шисдьсат` (шестьдесят), `тыcяч` (латиница вместо
кириллицы). Rule-based классификатор требует точного совпадения — это
**принципиальное ограничение**, исправляется только T5.

## 9. Известные ограничения

1. **Rule-based парсер не работает на ASR-шумных данных** (1.8% noisy
   accuracy) — это принципиальное ограничение, требует T5
2. **T5 не обучена на noisy** — clean-only 57%, noisy улучшит до ~68-75%
3. **real.f точность 55%** — пунктуация прилипает к токенам, требуется
   препроцессинг
4. **Noisy training прерван** на epoch 2, checkpoint утерян
5. **N-gram LM не работает** для confidence (длина mismatch:
   train 11.7 символов vs inference 150+)
6. **Только cardinal + ordinal** — нет дробей, дат, процентов, телефонов
7. **LoRA rank=8** — возможно, узкое горлышко (0.9M/66M = 1.36%)
