# Тестовое задание: Нормализация чисел в транскрибациях звонков Avito (NLP)

## Контекст задачи

В телефонных разговорах между менеджерами Авито и клиентами часто озвучиваются числовые значения:

- цены и бюджеты,
- сроки и длительности,
- количественные ограничения.

После автоматической расшифровки (ASR) такие числа попадают в текст в искаженном виде:

- с орфографическими ошибками
- с пропущенными буквами
- со склейками слов

Примеры фраз из транскрибаций:

- «бюджет гдето триста пятьдесят тысяч»
- «на двадцать три дня»
- «примерно пятнадцить показов»

Это затрудняет работу сразу по нескольким направлениям: ухудшает читаемость, затрудняет проверку фактов по диалогу, усложняет любые задачи анонимизации

## Цель задания

Разработать решение, которое:

- Находит в тексте все числовые выражения, записанные словами
- Устойчиво работает при наличии ошибок транскрибации
- Преобразует найденные выражения в нормализованное числовое значение

## Правила восстановления чисел

- Мы восстанавливаем любые числа, как количественные, так и порядковые
  - "первого числа" -> "1 числа"
  - "один день" -> "1 день"
- Мы группируем подряд идущие числительные, если это необходимо (логика сумм)
  - "да да да думаю две тысячи пятьсот это выйдет гдето " -> "да да да думаю 2500 это выйдет гдето "
- Мы не группируем перечисления одного порядка (логика надиктовки)
  - "да да да думаю двести триста выйдет гдето " -> "да да да думаю 200 300 это выйдет гдето "
- И помните, артефакты транскрибации могут вносить существенный шум. Не все корнер кейсы поддаются разумному обобщению и приближению. Это нормально

## Входные данные

Вам будут даны 2 файла в формате feather:

- **test.f** — 500 фрагментов транскрибаций без известной нормализации. Нормализация этих текстов и будет ответом на задание
- **calibration.f** — 500 фрагментов транскрибаций с заданной нормализации. Вы можете использовать его для предварительной оценки своего решения

Файлы можно скачать по ссылке: https://drive.google.com/drive/folders/1jgm9hGvZETTzhn4bu43e5UlI5gT12pud

- Колонка **"task_text"** содержит целевой фрагмент. Без капитализации, без пунктуации. Возможны ошибки распознавания. Есть в обоих файлах
  - Пример: "бюджет где то триста петьдесят тысич рублеи"
- Колонка **"ground_truth"** содержит нормализованный текст. Есть в файле calibration

## Целевая метрика

Решение будет оцениваться по метрике **Accuracy**: доля фраз после нормализации, которые соответствуют эталону к общему числу фраз

| task_text                                                       | answer                                                | ground_truth                                          | is_equal |
| --------------------------------------------------------------- | ----------------------------------------------------- | ----------------------------------------------------- | -------- |
| вот таким образом советую вам заложить бюджет в пятьдесят тысяч | вот таким образом советую вам заложить бюджет в 50000 | вот таким образом советую вам заложить бюджет в 50000 | True     |
| так с тридцати тысяч будет ставка около процентов двух          | так с 30000 будет ставка около процентов двух         | так с 30000 будет ставка около процентов 2            | False    |
| да да да думаю две тысячи пятьсот это выйдет гдето              | да да да думаю 2500 это выйдет гдето                  | да да да думаю 2500 это выйдет гдето                  | True     |
| да да да думаю двести триста выйдет гдето                       | да да да думаю 500 это выйдет гдето                   | да да да думаю 200 300 это выйдет гдето               | False    |
| давай с вами семнадцатого числа созвонимся тогда                | давай с вами 17 числа созвонимся тогда                | давай с вами 17 числа созвонимся тогда                | True     |

Accuracy = 3/5 \* 100 = 60%

Таким образом, для того, чтобы текст считался корректно нормализованным все числа в нем должны быть верно нормализованы. Искажение остального текста не допускается
